1️⃣ SQL: Your Data Query Language

SQL is the language for talking to structured data. No matter what DB you use — SQLite, PostgreSQL, DuckDB, BigQuery — the syntax is the same at the core.

You don't need to be a DBA. You do need to be fluent in SELECT, WHERE, JOIN, GROUP BY, and window functions. These five patterns cover 95% of what you'll do as an AI engineer.

The Five Essential SQL Patterns

SELECT & WHERE

Filter and project the columns and rows you need. The foundation of every query.

SELECT col FROM tbl WHERE condition

JOIN (INNER & LEFT)

Combine related tables. INNER = only matching rows. LEFT = keep all from left table.

FROM a JOIN b ON a.id = b.a_id

GROUP BY & HAVING

Aggregate rows into groups. HAVING filters groups (WHERE filters rows before grouping).

GROUP BY col HAVING COUNT(*) > 5

Window Functions

Rank, number, and aggregate across window frames without collapsing rows. Essential for AI data prep.

ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ...)

Subqueries & CTEs

Query within a query. CTEs (WITH clause) make complex queries readable. Used constantly in AI pipelines.

WITH ranked AS (SELECT ...) SELECT * FROM ranked WHERE rn = 1
▸ query.sql — All five patterns in action
-- Pattern 1: SELECT & WHERE — find recent high-value users SELECT id, name, email, total_spent, last_active FROM users WHERE total_spent > 100 AND last_active >= '2026-01-01' ORDER BY total_spent DESC LIMIT 100; -- Pattern 2: JOIN — enrich user data with order history SELECT u.name, o.id AS order_id, o.total, o.created_at FROM users u INNER JOIN orders o ON u.id = o.user_id WHERE o.created_at >= '2026-06-01'; -- Pattern 3: GROUP BY + HAVING — categories with lots of orders SELECT category, COUNT(*) AS order_count, SUM(total) AS revenue FROM orders o JOIN products p ON o.product_id = p.id GROUP BY category HAVING COUNT(*) > 10; -- Pattern 4: Window — rank products within each category SELECT name, category, price, ROW_NUMBER() OVER ( PARTITION BY category ORDER BY price DESC ) AS price_rank, RANK() OVER ( PARTITION BY category ORDER BY sales DESC ) AS sales_rank FROM products; -- Pattern 5: CTE — find top 3 products per category WITH ranked_products AS ( SELECT name, category, price, ROW_NUMBER() OVER ( PARTITION BY category ORDER BY price DESC ) AS rn FROM products ) SELECT * FROM ranked_products WHERE rn <= 3;
💡 AI engineer's note: The CTE + window function combo (Pattern 5) is the most useful pattern for AI pipelines. "Give me the top N items per category" — that's your RAG context building right there. Master this.

2️⃣ Data Cleaning with Pandas

Real-world data is always messy. Nulls, duplicates, wrong types, inconsistent formatting. Pandas is the standard tool for cleaning it up.

Handle Nulls

df.isnull(), df.dropna(), df.fillna(). Know when to drop vs. fill. Never ignore them.

Deduplication

df.duplicated(), df.drop_duplicates(). Always check before loading into a DB.

Type Conversion

df.astype(), pd.to_datetime(), pd.to_numeric(). Dates and numbers stored as strings will break your queries.

Merge DataFrames

pd.merge(df1, df2, on='key', how='inner'). The pandas equivalent of SQL JOIN.

▸ transform.py — Cleaning pipeline for AI
import pandas as pd # 1. Load raw data df = pd.read_csv("raw_orders.csv") print(df.info()) # See types, null counts print(df.head()) # Peek at values # 2. Handle nulls df = df.dropna(subset=["order_id", "user_id"]) # Can't recover these df["amount"] = df["amount"].fillna(float("nan")).astype("float64") df["category"] = df["category"].fillna("unknown") # 3. Type conversion df["created_at"] = pd.to_datetime(df["created_at"], errors="coerce") df["amount"] = pd.to_numeric(df["amount"], errors="coerce") # 4. Dedup df = df.drop_duplicates(subset=["order_id"]) # 5. Feature engineering for AI df["order_month"] = df["created_at"].dt.month df["is_weekend"] = df["created_at"].dt.dayofweek >= 5 # 6. Export clean data df.to_csv("clean_orders.csv", index=False) df.to_parquet("clean_orders.parquet", index=False) # Smaller, faster

3️⃣ File Formats: CSV vs. JSON vs. Parquet

You'll encounter three main formats. Each has a job.

Format Best For Pros Cons AI Pipeline Use
CSV Human-readable tabular data, small-medium datasets Universal, opens in Excel, easy to inspect No schema, big files are slow, no nested data Quick exports, debugging, sharing with non-engineers
JSON Nested/hierarchical data, API responses Flexible, self-describing, LLMs understand it natively Verbose, slower to parse, no columnar compression LLM I/O format, web API data, config files
Parquet Large datasets, columnar workloads, ML/analytics Fast, compressed, schema'd, columnar pruning Binary, not human-readable, requires tooling Production AI/ML pipelines, large-scale feature stores
▸ io_patterns.py — Reading & writing all three
import pandas as pd # CSV — universal, human-readable df = pd.read_csv("data.csv") df.to_csv("output.csv", index=False) # JSON — flexible, LLM-friendly df = pd.read_json("data.json") df.to_json("output.json", orient="records") # Parquet — fast, compressed, production-ready df = pd.read_parquet("data.parquet") df.to_parquet("output.parquet", compression="snappy") # Rule of thumb: # - During development: CSV (inspect easily) # - LLM context building: JSON (models parse it well) # - Production pipelines: Parquet (speed & compression)

4️⃣ ETL Patterns for AI & RAG

ETL (Extract, Transform, Load) is the backbone of every data pipeline. For AI engineers, it's how you build the data layer that feeds your models.

E Extract — Get data from source: CSV files, JSON APIs, SQL databases, web scraping. Often pull incrementally (only new/changed records).
T Transform — Clean, normalize, chunk, embed. This is where pandas shines. For RAG: split documents into chunks, generate embeddings, store metadata.
L Load — Write to target: SQLite for dev, PostgreSQL for prod, vector DB for embeddings. The format must match what your application expects.
▸ etl_pipeline.py — Minimal RAG ETL
import pandas as pd import sqlite3 # ── EXTRACT ── raw = pd.read_json("documents.json") # documents with 'id', 'title', 'body', 'source' # ── TRANSFORM ── # Clean clean = raw.dropna(subset=["id", "body"]) clean = clean.drop_duplicates(subset=["id"]) clean["body"] = clean["body"].str.strip() # Chunk for RAG (simple character split) chunks = [] for _, row in clean.iterrows(): body = row["body"] for i in range(0, len(body), 500): chunks.append({ "doc_id": row["id"], "title": row["title"], "chunk_index": i // 500, "chunk_text": body[i:i+500], "source": row["source"], }) chunk_df = pd.DataFrame(chunks) # ── LOAD ── conn = sqlite3.connect("rag_store.db") chunk_df.to_sql("chunks", conn, if_exists="replace", index=False) conn.close() print(f"Loaded {len(chunk_df)} chunks into rag_store.db")

⚠️ Pitfalls — Read These Before the Exercises

Pitfall 1: Thinking ORMs replace SQL
ORMs (like SQLAlchemy, Django ORM) are great for CRUD apps. They're awful for analytical queries. JOINs with aggregations and window functions written in an ORM produce unreadable, slow SQL. Fix: Write raw SQL for anything beyond simple CRUD. Your ORM is for writes. SQL is for questions.
Pitfall 2: Ignoring NULL handling
NULL + anything = NULL. Most bugs in SQL queries come from forgetting that NULLs propagate. WHERE col != 'x' won't return rows where col IS NULL. Fix: Always check your WHERE clauses for NULL exclusion. Use COALESCE() or IS NULL explicitly.
Pitfall 3: Wrong JOIN type
Using INNER JOIN when you need LEFT JOIN means silently dropping rows. Using LEFT JOIN when you need INNER JOIN means getting unexpected NULLs. Fix: Ask yourself: "Do I want ALL rows from the left table, even if they don't match in the right one?" If yes → LEFT JOIN. If only matches → INNER JOIN.
Pitfall 4: Loading everything into memory
Pandas loads data into RAM. If your CSV is 20GB and you have 16GB of RAM, you crash. Fix: Use pd.read_csv(..., chunksize=10000), or switch to polars/pyarrow/dask for larger-than-memory data. For production, Parquet + column pruning is your friend.
Pitfall 5: Not checking for duplicates
Your AI application asks "how many orders?" and gets an inflated number because the raw CSV had duplicate rows. Every analytical query is wrong by the same factor. Fix: Always df.duplicated().sum() before loading into any database or generating any aggregate statistic.

🧠 Key Takeaways

Your new default mental model:
Every AI system is a data pipeline with an LLM in the middle. The quality of your input data determines the quality of your output. SQL asks the questions. Pandas cleans the mess. ETL connects the dots.
  • SQL is your data query language — SELECT, JOIN, GROUP BY, window functions, CTEs
  • Pandas is your data cleaning workbench — handle nulls, dedup, type conversion, merge
  • File formats — CSV for dev, JSON for LLMs, Parquet for production
  • ETL (Extract → Transform → Load) is the pattern that powers every RAG pipeline
  • The top N per category pattern (CTE + ROW_NUMBER + WHERE rn <= N) is your most reusable query
  • Pitfalls to never forget: NULL propagation, wrong JOIN type, ORM-for-SQL substitution, in-memory limits, unchecked duplicates
