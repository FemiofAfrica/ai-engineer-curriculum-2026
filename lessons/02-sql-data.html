Why Learn SQL?

Every AI system works with data. The model you call with an API is only the last 10% — the real work is getting the right data into it and handling what comes out.

SQL is how you talk to databases. It's been around for 50 years and it's not going anywhere. Every AI engineer — whether they're building RAG pipelines, analyzing datasets, or fine-tuning models — needs to be comfortable with SQL.

The best part? You already know the most important concept: asking questions. SQL is just a language for asking questions of structured data.

1️⃣ What is a Database?

A database is a place to store structured data in an organized way. Think of it like a smart filing cabinet — you put data in, you take data out, and you can find exactly what you need without digging through everything.

📎 Think of it like Excel

If you've ever used Excel or Google Sheets, you already understand databases at a high level:

  • An Excel workbook = a database (a file that holds everything)
  • A sheet/tab = a table (one structured set of data, like "Movies" or "Customers")
  • A row = a record (one item, like one movie)
  • A column = a field (one piece of info, like the movie's title or year)

The difference? Databases are faster, more powerful, handle way more data, and let you ask complex questions across multiple tables at once.

Here's what a simple table called movies might look like:

id title year rating
1The Matrix19998.7
2Inception20108.8
3Interstellar20148.6

Each row is one movie. Each column is one attribute of that movie. Simple, right?

2️⃣ What is SQL?

SQL stands for Structured Query Language (pronounced "sequel" or "ess-queue-ell" — both are fine).

It's the language you use to talk to a database. You write a query — a question — and the database answers with data.

SELECT

"Show me data from this table"

WHERE

"Only include rows that match this condition"

INSERT

"Add new data to this table"

UPDATE

"Change existing data"

DELETE

"Remove data from this table"

CREATE TABLE

"Define a new table structure"

These six operations cover 95% of what you'll do day-to-day. You don't need to memorize everything at once — you'll learn by doing.

🔁 How SQL Works

CREATE TABLE → INSERT Data → SELECT / WHERE → UPDATE → DELETE

3️⃣ Getting Started with SQLite

We'll use SQLite for this module. Why?

  • No server needed — the database is just a single file on your computer
  • Pre-installed on Mac — you already have it
  • Same SQL — everything you learn works with PostgreSQL, MySQL, DuckDB, etc.
  • What you learn here applies everywhere — SQL is SQL

Check if SQLite is installed

Open your terminal and run:

terminal
$ sqlite3 --version 3.43.2 2024-10-10 16:45:00

If you see a version number, you're all set. If not, install it with Homebrew:

terminal
$ brew install sqlite3

Create your first database

A SQLite database is just a file. To create one (or open an existing one), run:

terminal
$ sqlite3 my-first-db.db SQLite version 3.43.2 2024-10-10 16:45:00 Enter ".help" for usage hints. sqlite>

That sqlite> prompt means you're inside SQLite. You can now type SQL commands. Type .quit to exit.

4️⃣ Creating Your First Table

A table needs a structure before you can put data in it. You define that structure with CREATE TABLE.

create_movies.sql 
-- Create a table called 'movies'
CREATE TABLE movies (
    id      INTEGER PRIMARY KEY,  -- unique ID for each movie
    title   TEXT,                  -- the movie title (text)
    year    INTEGER,               -- release year (whole number)
    rating  REAL                   -- rating out of 10 (decimal)
);

Let's break down what each part means:

Part Meaning
INTEGER A whole number (like 1999, 42, 0)
TEXT Letters and words (like "The Matrix")
REAL A decimal number (like 8.7, 3.14)
PRIMARY KEY Marks this column as the unique identifier — no two rows can have the same value

Copy-paste the CREATE TABLE statement at the sqlite> prompt:

sqlite3 my-first-db.db
sqlite> CREATE TABLE movies ( ...> id INTEGER PRIMARY KEY, ...> title TEXT, ...> year INTEGER, ...> rating REAL ...> ); sqlite>

If you don't see an error, it worked! The table now exists in your database. Run .tables to confirm:

sqlite3 my-first-db.db
sqlite> .tables movies

5️⃣ Adding Data

An empty table isn't very useful. Let's add some movies with INSERT.

insert_movies.sql 
INSERT INTO movies VALUES (1, 'The Matrix', 1999, 8.7);
INSERT INTO movies VALUES (2, 'Inception', 2010, 8.8);
INSERT INTO movies VALUES (3, 'Interstellar', 2014, 8.6);
INSERT INTO movies VALUES (4, 'The Godfather', 1972, 9.2);
INSERT INTO movies VALUES (5, 'Parasite', 2019, 8.5);
INSERT INTO movies VALUES (6, 'Spirited Away', 2001, 8.6);

Type each line at the sqlite> prompt. The order of values must match the order of columns you defined: (id, title, year, rating).

💡 Important: Text values must be in single quotes: 'The Matrix'. Numbers don't need quotes.

6️⃣ Asking Questions with SELECT

SELECT is the most important SQL command. You use it to get data out of a database.

Get everything

The simplest query — get all columns and all rows:

query.sql 
SELECT * FROM movies;

The * means "all columns". Let's see what comes back:

sqlite3 my-first-db.db
sqlite> SELECT * FROM movies; 1|The Matrix|1999|8.7 2|Inception|2010|8.8 3|Interstellar|2014|8.6 4|The Godfather|1972|9.2 5|Parasite|2019|8.5 6|Spirited Away|2001|8.6

Get specific columns

Instead of *, list the columns you want:

query.sql 
SELECT title, year FROM movies;
sqlite3 my-first-db.db
sqlite> SELECT title, year FROM movies; The Matrix|1999 Inception|2010 Interstellar|2014 The Godfather|1972 Parasite|2019 Spirited Away|2001

Filter with WHERE

The real power of SQL is filtering. Use WHERE to only get the rows you care about.

query.sql 
-- Movies after the year 2000
SELECT title, year FROM movies WHERE year > 2000;

-- Movies with rating 8.7 or higher
SELECT title, rating FROM movies WHERE rating >= 8.7;

-- Exactly one movie by title
SELECT * FROM movies WHERE title = 'Inception';

-- Combine conditions
SELECT * FROM movies
WHERE year > 2000 AND rating >= 8.6;
sqlite3 my-first-db.db
sqlite> SELECT title FROM movies WHERE year > 2000; Inception Interstellar Parasite Spirited Away sqlite> SELECT title, rating FROM movies WHERE rating >= 8.7; The Matrix|8.7 Inception|8.8 The Godfather|9.2 sqlite> SELECT * FROM movies WHERE title = 'Inception'; 2|Inception|2010|8.8

Comparison operators you can use in WHERE:

OperatorMeaningExample
=Equal toWHERE title = 'The Matrix'
>Greater thanWHERE year > 2000
<Less thanWHERE year < 2010
>=Greater than or equalWHERE rating >= 8.0
<=Less than or equalWHERE year <= 2001
!= or <>Not equal toWHERE title != 'Inception'

7️⃣ Updating and Deleting

UPDATE — change existing data

update.sql 
-- ⚠️ Update Parasite's rating (note the WHERE!)
UPDATE movies SET rating = 8.6 WHERE title = 'Parasite';

-- Change multiple columns at once
UPDATE movies
SET rating = 9.0, year = 2010
WHERE title = 'Inception';
🔥 CRITICAL: Never forget WHERE on UPDATE!
UPDATE movies SET rating = 10.0; — without WHERE, this changes EVERY ROW. Every movie's rating becomes 10.0. Always double-check your WHERE clause before hitting Enter.

DELETE — remove data

delete.sql 
-- ⚠️ Delete one movie (note the WHERE!)
DELETE FROM movies WHERE title = 'Spirited Away';

-- Delete all movies with rating below 8.0
DELETE FROM movies WHERE rating < 8.0;
🔥 CRITICAL: Never forget WHERE on DELETE!
DELETE FROM movies; — without WHERE, this deletes EVERY ROW in the table. There is no undo (unless you have a backup). Always, always use WHERE with DELETE.
💡 Pro tip: When testing an UPDATE or DELETE, first write it as a SELECT to see which rows will be affected:
SELECT * FROM movies WHERE rating < 8.0; ← runs this first, then change SELECT * to DELETE.

8️⃣ Sorting and Limiting

Use ORDER BY to sort results and LIMIT to get only a certain number of rows.

sorting.sql 
-- Movies sorted by year, oldest first
SELECT title, year FROM movies ORDER BY year;

-- Movies sorted by rating, highest first
SELECT title, rating FROM movies ORDER BY rating DESC;

-- Only the top 3 highest-rated movies
SELECT title, rating FROM movies
ORDER BY rating DESC
LIMIT 3;

-- Latest 2 movies (by year, descending, take top 2)
SELECT title, year FROM movies
ORDER BY year DESC
LIMIT 2;
sqlite3 my-first-db.db
sqlite> SELECT title, rating FROM movies ORDER BY rating DESC LIMIT 3; The Godfather|9.2 Inception|9.0 The Matrix|8.7

By default, ORDER BY sorts ascending (A-Z, 0-9). Add DESC for descending (Z-A, 9-0).

9️⃣ Grouping and Counting

Sometimes you don't want individual rows — you want summary information. That's where GROUP BY comes in.

Let's add a "genre" column to make grouping more interesting:

alter.sql 
-- Add a genre column to our movies table
ALTER TABLE movies ADD COLUMN genre TEXT;

-- Update with genre values
UPDATE movies SET genre = 'Sci-Fi' WHERE title IN ('The Matrix', 'Inception', 'Interstellar');
UPDATE movies SET genre = 'Drama' WHERE title = 'The Godfather';
UPDATE movies SET genre = 'Thriller' WHERE title = 'Parasite';
UPDATE movies SET genre = 'Animation' WHERE title = 'Spirited Away';

Now let's ask some grouped questions:

group_by.sql 
-- How many movies per genre?
SELECT genre, COUNT(*) AS count
FROM movies
GROUP BY genre;

-- What's the average rating per genre?
SELECT genre, AVG(rating) AS avg_rating
FROM movies
GROUP BY genre;

-- How many movies per decade?
SELECT year / 10 * 10 AS decade, COUNT(*) AS count
FROM movies
GROUP BY decade;
sqlite3 my-first-db.db
sqlite> SELECT genre, COUNT(*) AS count FROM movies GROUP BY genre; Animation|1 Drama|1 Sci-Fi|3 Thriller|1

COUNT(*) counts how many rows in each group. Other useful functions: AVG() (average), SUM() (total), MIN(), MAX().

🔗 Simple JOINs

So far we've worked with one table. Real databases have many tables that connect to each other.

Let's create a second table — directors — and connect it to our movies:

directors.sql 
CREATE TABLE directors (
    id   INTEGER PRIMARY KEY,
    name TEXT
);

INSERT INTO directors VALUES (1, 'Lana Wachowski');
INSERT INTO directors VALUES (2, 'Christopher Nolan');
INSERT INTO directors VALUES (3, 'Francis Ford Coppola');
INSERT INTO directors VALUES (4, 'Bong Joon-ho');
INSERT INTO directors VALUES (5, 'Hayao Miyazaki');

Now we need a way to say "this movie was directed by this director." We'll add a director_id column to our movies table:

link.sql 
ALTER TABLE movies ADD COLUMN director_id INTEGER;

UPDATE movies SET director_id = 1 WHERE title = 'The Matrix';
UPDATE movies SET director_id = 2 WHERE title IN ('Inception', 'Interstellar');
UPDATE movies SET director_id = 3 WHERE title = 'The Godfather';
UPDATE movies SET director_id = 4 WHERE title = 'Parasite';
UPDATE movies SET director_id = 5 WHERE title = 'Spirited Away';

Now the movies table has a director_id column that refers to the id column in the directors table. This is called a foreign key — it links rows across tables.

To get information from both tables at once, we use JOIN:

join.sql 
-- Combine movies and directors — match rows where the IDs align
SELECT movies.title, directors.name
FROM movies
INNER JOIN directors ON movies.director_id = directors.id;
sqlite3 my-first-db.db
sqlite> SELECT movies.title, directors.name ...> FROM movies ...> INNER JOIN directors ON movies.director_id = directors.id; The Matrix|Lana Wachowski Inception|Christopher Nolan Interstellar|Christopher Nolan The Godfather|Francis Ford Coppola Parasite|Bong Joon-ho Spirited Away|Hayao Miyazaki

🧩 How JOIN works

Think of it as looking up a value in one table and finding its match in another:

  1. Take a row from the movies table
  2. Look at its director_id (e.g., 2)
  3. Find the row in directors where id = 2
  4. Combine them into one big row — movie data + director data

That's what ON movies.director_id = directors.id does. It matches rows where the IDs are equal.

You can also use a shorter form with table aliases:

join_short.sql 
SELECT m.title, d.name
FROM movies m
INNER JOIN directors d ON m.director_id = d.id
WHERE m.year > 2000
ORDER BY m.rating DESC;

📤 Exporting from SQL

Once you've queried your data, you'll often want to export it to a file. SQLite has built-in commands for this.

sqlite3 my-first-db.db
-- Export query results to CSV sqlite> .mode csv sqlite> .output movies.csv sqlite> SELECT * FROM movies; sqlite> .output stdout -- Now check the file sqlite> .quit $ cat movies.csv

Step by step:

  1. .mode csv — tell SQLite to output in CSV format
  2. .output movies.csv — send output to a file instead of the screen
  3. Run your SELECT query
  4. .output stdout — send output back to the screen

The resulting CSV file can be opened in Excel, imported into Python with pandas, or shared with anyone.

⚠️ Pitfalls — Read These Before the Exercises

🔥 Forgetting WHERE on DELETE or UPDATE
DELETE FROM movies; deletes every single row. UPDATE movies SET rating = 0; changes every row. Always write your WHERE clause first, then compose the rest of the statement.
🚨 Missing semicolons
SQL statements need a semicolon (;) at the end. Without it, SQLite will just show you a continuation prompt (...>) and wait. Type the semicolon and hit Enter to execute.
🧨 Confusing = in SQL vs == in Python
In SQL, WHERE title = 'The Matrix' uses a single = for comparison. In Python, you'd write title == 'The Matrix' with double ==. SQL doesn't use ==.
💣 INSERTing text without quotes
INSERT INTO movies VALUES (1, The Matrix, 1999, 8.7); will fail because The Matrix isn't quoted. Text must be in single quotes: 'The Matrix'.
🔑 Using double quotes instead of single quotes
SQLite uses single quotes for strings: 'The Matrix'. Double quotes ("The Matrix") are for database identifiers (table names, column names). Mixing them up causes confusing errors.
📏 Not closing the database connection
When you're done in SQLite, type .quit to exit. If you just close the terminal window, any uncommitted changes could be lost. SQLite auto-commits by default, but it's a good habit to exit properly.
