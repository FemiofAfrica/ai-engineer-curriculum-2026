End-to-End Embedding Search Architecture

What Embeddings Are

Embeddings are dense vector representations of semantic meaning — an array of floating-point numbers that captures what a piece of text means rather than what words it contains.

🔑 The Key Insight

"The cat sat on the mat" and "A feline rested on the rug" produce vectors that are close together in semantic space, even though they share almost no keywords. A search for "feline" can match "cat" because of their vector proximity — not because of string matching.

An embedding model maps each input (word, sentence, paragraph, document) to a fixed-size vector — typically 384 to 3072 dimensions. Each dimension encodes a learned semantic feature, and the vector's position in this high-dimensional space represents the text's meaning.

🧬

Dense Vectors

Unlike sparse one-hot encodings (bag-of-words), embeddings are dense — every dimension carries information. A 1536-dim vector from text-embedding-3-small captures nuanced semantics in compact form.

🌌

Semantic Space

Vectors live in a high-dimensional space where distance ≈ semantic difference. "King" − "Man" + "Woman" ≈ "Queen" — the classic analogy works because of vector arithmetic in embedding space.

📏

Dimensions

More dimensions = more capacity to encode meaning, but also more storage and slower search. text-embedding-3-small (1536 dims) vs text-embedding-3-large (3072) — choose based on your precision needs vs cost/speed.

Embedding Models & APIs

🤖

OpenAI text-embedding-3

1536 dims (small) 3072 dims (large)

API-based. Most widely used. text-embedding-3-small is the cost-performance sweet spot at ~$0.02/1M tokens. Supports dimensionality reduction via the dimensions parameter — truncate to 256 dims for cheaper storage without losing much quality.

✅ Best for: Production RAG, cost-sensitive apps

📦

Sentence-Transformers

384 dims Local

Open-source, run locally. all-MiniLM-L6-v2 is the go-to: 384 dimensions, fast inference on CPU, good-enough quality for many use cases. all-mpnet-base-v2 is higher quality but slower. No API costs, no rate limits.

✅ Best for: Local dev, prototyping, privacy-sensitive data

🔗

Cohere Embed

1024 dims Multilingual

API-based with strong multilingual support. embed-english-v3.0 and embed-multilingual-v3.0. Distinctive feature: compression — returns both a full-precision vector and a smaller binary/short vector for cheaper retrieval.

✅ Best for: Multilingual apps, enterprise

embed.py — Generate embeddings with OpenAI and sentence-transformers 
import os
from openai import OpenAI

# === OpenAI embedding ===
client = OpenAI(api_key=os.getenv("OPENAI_API_KEY"))

text = "The cat sat on the mat."
response = client.embeddings.create(
    model="text-embedding-3-small",
    input=text,
    dimensions=256  # optional truncation
)
embedding = response.data[0].embedding
print(f"Vector dims: {len(embedding)}")   # 256
print(embedding[:5])                        # [-0.023, 0.045, ...]

# === Sentence-transformers (local) ===
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("all-MiniLM-L6-v2")
vec = model.encode(text)
print(f"Local vector: {len(vec)} dims")     # 384

Vector Databases

A vector database stores embeddings and supports fast similarity search by indexed nearest-neighbour algorithms. You don't need a dedicated vector DB — three of the four options here are extensions or libraries.

🐘

pgvector

Postgres Open Source

PostgreSQL extension that adds vector data type and IVFFlat / HNSW indexes. Zero new infrastructure — your existing Postgres becomes a vector DB. Supports SQL queries that mix vector search with traditional WHERE filters.

✅ Best for: Teams already on Postgres, hybrid SQL + vector queries

🌲

Pinecone

Managed Serverless

Fully-managed vector database. No infrastructure to manage. Automatic scaling, multi-tenancy, namespaces, metadata filtering. Uses HNSW by default. The easiest way to get started — but costs add up at scale.

✅ Best for: Fastest setup, production scale without ops

💎

Chroma

Local Lightweight

Embedding database designed for AI apps. Runs in-process (Python) or as a server. Simple API, zero config, DuckDB-backed for persistence. Perfect for prototyping and small-to-medium datasets.

✅ Best for: Prototyping, local dev, small datasets

⚡

Qdrant

Rust High Perf

Written in Rust — very fast. Supports filtering, payload storage, and multiple index types (HNSW, IVF). Can run as a binary, Docker container, or managed cloud. Good balance of performance and feature completeness.

✅ Best for: High-performance, filtering-heavy queries

index.py — Store embeddings in Chroma 
import chromadb
from sentence_transformers import SentenceTransformer

# Init
embed_model = SentenceTransformer("all-MiniLM-L6-v2")
client = chromadb.Client()
collection = client.create_collection(
    name="my_docs",
    metadata={ "hnsw:space": "cosine" }
)

# Documents to index
docs = [
    "The cat sat on the mat.",
    "Dogs love to play fetch in the park.",
    "A car drove down the highway at high speed.",
    "Quantum mechanics describes the behaviour of particles.",
]
ids = [f"doc{i}" for i in range(len(docs))]

# Embed + store
embeddings = embed_model.encode(docs).tolist()
collection.add(ids=ids, embeddings=embeddings, documents=docs)

print(f"Indexed {len(docs)} documents")

Similarity Metrics

Once you have vectors, you need a way to measure how close they are. Three main metrics, each with a different interpretation:

Cosine Similarity

cos(θ) = (A·B) / (||A|| × ||B||)

Measures the angle between vectors, ignoring magnitude. Range: [-1, 1]. Most common for text embeddings — you care about direction (meaning), not length (frequency).

🏆 Use when: comparing semantic meaning, text embeddings

Dot Product

A·B = Σ Aᵢ × Bᵢ

Un-normalised similarity that includes magnitude. When all vectors are unit-normalised, dot product = cosine similarity. Faster to compute than cosine.

🏆 Use when: vectors are normalised, speed matters in production

Euclidean (L2) Distance

d = √Σ(Aᵢ − Bᵢ)²

Geometric distance in vector space. Sensitive to magnitude — two similar ideas expressed at different lengths could appear far apart. Good for image embeddings or when magnitude matters.

🏆 Use when: magnitude carries information (image embeddings, audio)

⚡ Rule of Thumb

Text embeddings → use cosine similarity. It's the standard for semantic search. Most vector databases default to cosine and many normalise vectors internally, making dot product equivalent. L2 is useful when you're dealing with embeddings where magnitude carries signal (e.g., some multimodal models).

Indexing Algorithms: Speed vs. Accuracy

Brute-force search over every vector (flat search) is O(n). For millions of vectors, you need approximate nearest neighbour (ANN) indexes. Here are the trade-offs:

🔍 Flat Search

🐢 Slower

Brute-force — compares every vector. 100% recall. Use for small datasets (<10K vectors) or when perfect accuracy is non-negotiable.

🧱 IVF (Inverted File Index)

⚡ Moderate

Clusters vectors into groups (Voronoi cells). At query time, searches only the nearest cells. Parameter nprobe controls speed/accuracy trade-off. Good balance for up to ~1M vectors.

🌐 HNSW (Hierarchical NSW)

🚀 Fastest

Builds a multi-layer graph. At query time, navigates from coarse to fine layers. Best-in-class speed/accuracy for large datasets. Higher memory usage. Industry standard for production vector search.

IndexBuild TimeQuery SpeedMemoryRecallBest For
Flat (brute force)NoneO(n)Low100%Small datasets, validation
IVFFlatFastFast*Low~90-98%Medium datasets, limited RAM
HNSWSlowVery fastHigh~99%+Large datasets, production

⚠️ The Index Build Trap

HNSW takes time to build. The graph construction is O(n log n) and memory-intensive. For 1M 1536-dim vectors, building an HNSW index can take 10–30 minutes on a typical machine. Plan for this — don't expect instant indexing of large datasets. IVFFlat builds much faster but queries are slower at high recall.

search.py — Semantic search with Chroma 
# Continue from index.py — query the collection
query = "animals that sit on furniture"
query_vec = embed_model.encode(query).tolist()

results = collection.query(
    query_embeddings=[query_vec],
    n_results=2
)

for i, doc in enumerate(results["documents"][0]):
    dist = results["distances"][0][i]
    print(f"[{dist:.4f}] {doc}")

# Output:
# [0.2345] The cat sat on the mat.
# [0.5123] Dogs love to play fetch in the park.

Hybrid Search: Keyword + Vector

🔑 The Core Insight

Vector search alone is not enough. For queries that contain a specific term (e.g., "Python 3.12 async error handling"), a keyword match on "Python" or "async" may be more relevant than semantic neighbours. Hybrid search combines BM25 (keyword) scores with vector similarity scores, then re-ranks.

Pure vector search handles semantic matching well but can miss exact-term precision. Pure keyword search (BM25, TF-IDF) is great for exact terms but misses synonyms and paraphrases. Hybrid search combines both — typically via reciprocal rank fusion (RRF) or weighted averaging.

📖

BM25 (Keyword)

Term-based ranking. Great for exact matches, IDs, code snippets, proper nouns. Sparse — most dimensions are zero. Not affected by embedding quality or domain shift.

🏆 Excels at: Exact term matching, code search, acronyms

🧠

Vector (Semantic)

Dense embedding search. Captures meaning, synonyms, paraphrases. Generalises beyond keyword overlap. Weak on exact terms, rare words, and domain-specific jargon not seen in training.

🏆 Excels at: Synonyms, paraphrases, conceptual similarity

🔀

Reciprocal Rank Fusion (RRF)

Combines two ranked lists by computing RRF(d) = 1/(k + r₁(d)) + 1/(k + r₂(d)) where r₁ and r₂ are ranks from each method. Simple, effective, no training needed. The de facto hybrid search method.

🏆 Excels at: Combining heterogenous rankers

hybrid.py — Combine BM25 + vector search with RRF 
import math
from rank_bm25 import BM25Okapi

# BM25 index (on tokenised documents)
tokenised = [doc.lower().split() for doc in docs]
bm25 = BM25Okapi(tokenised)

def rrf_hybrid(query, k=60, top_n=5):
    # 1. Get BM25 scores
    bm25_scores = bm25.get_scores(query.lower().split())
    bm25_ranks = sorted(
        range(len(bm25_scores)),
        key=lambda i: bm25_scores[i],
        reverse=True
    )

    # 2. Get vector scores (cosine similarity)
    query_vec = embed_model.encode(query)
    vec_scores = [
        cosine_similarity(query_vec, doc_vec)
        for doc_vec in doc_embeddings
    ]
    vec_ranks = sorted(
        range(len(vec_scores)),
        key=lambda i: vec_scores[i],
        reverse=True
    )

    # 3. Reciprocal Rank Fusion
    rrf_scores = {}
    for i in range(len(docs)):
        r1 = bm25_ranks.index(i) + 1
        r2 = vec_ranks.index(i) + 1
        rrf_scores[i] = 1/(k+r1) + 1/(k+r2)

    return sorted(
        rrf_scores.items(),
        key=lambda x: x[1],
        reverse=True
    )[:top_n]

# Usage
query = "car on the road"
results = rrf_hybrid(query)
for idx, score in results:
    print(f"[RRF {score:.4f}] {docs[idx]}")

⛔ Pitfalls — Read Before You Build

🚫 Treating All Dimensions Equally

Not all embedding dimensions carry the same signal. Some dimensions encode broad semantic axes (e.g., "positive vs negative sentiment"), others encode noise. Dimensionality reduction (via the dimensions parameter in OpenAI's API) or PCA can strip noise and speed up search. Experiment with 256 or 512 dims before defaulting to the full vector.

🚫 Ignoring Index Build Time

HNSW indexes take minutes, not seconds to build on large datasets. If you're prototyping iteratively, use flat search first (small data), then switch to HNSW when you're ready for production. Don't block your dev loop on index construction — and never surprise yourself with a 30-minute build time on deploy day.

🚫 Assuming Vector Search Alone Is Enough

Vector search is powerful, but it fails on exact-term queries, rare entities, domain-specific terminology not in the training data, and queries where keyword overlap is the strongest signal. Always benchmark hybrid search (BM25 + vector) against pure vector on your data. In most real-world applications, hybrid wins.

🚫 Not Normalising Embeddings

Different models produce vectors at different scales. If you use cosine similarity but don't normalise your vectors, the results are meaningless. Some vector DBs normalise internally (Chroma does for cosine); others (pgvector) don't. Always check your DB's default behaviour and normalise explicitly (vec / ||vec||) if needed.

🚫 Forgetting Metadata Filtering

In production, you almost never search the whole vector space — you search within a user's documents, a date range, or a content category. If your vector DB supports metadata filtering (Pinecone, Qdrant, pgvector), use it. Filtering before KNN search can 10× reduce query latency.
