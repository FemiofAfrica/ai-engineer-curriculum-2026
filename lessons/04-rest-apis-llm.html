REST APIs & LLM Integration

The central skill of the AI engineer: wiring models into your applications. Master HTTP, API authentication, streaming, error handling, and the OpenAI & Anthropic APIs — the two providers you'll use most.

📅 Phase 1 · Week 4 · ~5–7 days

Client → API Call → LLM Provider → Streaming Response → Client

📤 Request → 🔑 Auth → ⏱️ Rate Limit → 📡 Stream → 🔍 Parse → 🛡️ Error Handle

1. HTTP Basics for API Integration

Every LLM API call is an HTTP request. If you understand GET vs POST, status codes, and authentication, you already understand most of what the SDKs are doing under the hood.

GET vs POST — LLM APIs use POST because you're sending a large payload (your messages/prompt) and the request is not idempotent. GET requests send data in the URL query string (limited length); POST sends a JSON body that can hold megabytes of conversation history.

Headers — Every API call needs:

  • Authorization: Bearer sk-... — your API key
  • Content-Type: application/json — the body format
  • Accept: text/event-stream — for streaming responses

Status codes you'll see constantly:

200 OK — success
400 Bad request
401 Unauthorized
429 Rate limited
500 Server error
🌐

HTTP Methods & Status Codes

  • GET Read data (not for LLMs)
  • POST Send prompt, get completion
  • 200 — success, body contains response
  • 400 — invalid request (bad messages format)
  • 401 — bad or missing API key
  • 429 — rate limit exceeded (you'll see this a lot)
  • 500 — provider-side error (retryable)
🤖

OpenAI API

  • Endpoint: POST /v1/chat/completions
  • Messages format: {role, content} — system, user, assistant
  • Key params: temperature, max_tokens, top_p
  • Streaming: SSE via stream: true
  • Function calling: tools parameter
  • SDK: openai (official Python package)
🧠

Anthropic API

  • Endpoint: POST /v1/messages
  • System prompt: separate system parameter
  • Messages format: user + assistant alternating
  • Streaming: SSE with stream: true
  • Tool use: tools parameter
  • Thinking mode: thinking: {type: "enabled"}
  • SDK: anthropic (official Python package)
🛡️

Error Handling

  • Rate limits (429): exponential backoff + jitter
  • Token limit errors: truncate messages, reduce max_tokens
  • Timeouts: set reasonable timeout on requests
  • Server errors (500): retry with backoff (max 3 attempts)
  • Auth errors (401): don't retry — check your key
🏗️

API Design Patterns

  • Endpoints: single base URL per provider
  • Auth: Bearer token in Authorization header
  • Versioning: /v1/ prefix on all endpoints
  • Env vars: OPENAI_API_KEY, ANTHROPIC_API_KEY
  • .env files: never hardcode keys in source

2. OpenAI API — Chat Completions

The most-called API in AI engineering. One endpoint, one message format, infinite possibilities. Let's break it down.

The chat/completions endpoint accepts a list of messages (each with role and content), model name, and parameters. It returns a completion. Here's the minimal call:

llm_client.py 
from openai import OpenAI
import os

client = OpenAI(api_key=os.getenv("OPENAI_API_KEY"))

response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "Explain REST APIs like I'm 12."},
    ],
    temperature=0.7,
    max_tokens=500,
)

print(response.choices[0].message.content)

Key insight: The temperature parameter controls randomness (0 = deterministic, 1 = creative). max_tokens limits the response length. Both are critical for production — set them explicitly every time.

📡 Streaming with SSE

Streaming (Server-Sent Events) sends tokens one-by-one as the model generates them. Without streaming, you wait for the full response. With streaming, you show tokens as they arrive — critical for UX.

streaming.py 
from openai import OpenAI

client = OpenAI()

stream = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "Count from 1 to 5."}],
    stream=True,        # ← enables streaming
)

full_response = ""
for chunk in stream:
    delta = chunk.choices[0].delta
    if delta.content:
        full_response += delta.content
        print(delta.content, end="", flush=True)

print(f"\n\nFull response ({len(full_response)} chars)")

🔧 Function & Tool Calling

OpenAI's function calling lets the model decide to call a function you define. It doesn't execute the function — it returns a structured JSON argument. You execute it. This is the foundation of agents.

function_calling.py 
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "What's the weather in Lagos?"}],
    tools=[{
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "Get weather for a city",
            "parameters": {
                "type": "object",
                "properties": {
                    "location": {"type": "string"}
                },
                "required": ["location"]
            }
        }
    }],
    tool_choice="auto"
)

msg = response.choices[0].message
if msg.tool_calls:
    print(f"Model wants to call: {msg.tool_calls[0].function.name}")
    print(f"Arguments: {msg.tool_calls[0].function.arguments}")

3. Anthropic API — Messages

Anthropic's API has a different philosophy — a separate system prompt, alternating user/assistant messages, and a rich tool-use system.

Unlike OpenAI, the system prompt is a top-level parameter, not a message role. Messages alternate: user → assistant → user → assistant.

anthropic_client.py 
from anthropic import Anthropic
import os

client = Anthropic(api_key=os.getenv("ANTHROPIC_API_KEY"))

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    system="You are a concise, helpful assistant.",  # ← top-level
    messages=[
        {"role": "user", "content": "What's the difference between REST and GraphQL?"}
    ],
    max_tokens=500,
    temperature=0.7,
)

print(response.content[0].text)

🔧 Tool Use (Anthropic's Function Calling)

Anthropic calls it "tool use." Same concept as OpenAI's function calling: define tools with JSON Schema, the model returns a call, you execute it.

anthropic_tools.py 
response = client.messages.create(
    model="claude-sonnet-4-20250514",
    system="You can use the available tools to answer questions.",
    messages=[{"role": "user", "content": "Search for AI Engineer jobs in Lagos"}],
    tools=[{
        "name": "search_jobs",
        "description": "Search job listings",
        "input_schema": {       # ← note: input_schema, not parameters
            "type": "object",
            "properties": {
                "query": {"type": "string"},
                "location": {"type": "string"}
            },
            "required": ["query"]
        }
    }],
    max_tokens=1000
)

for block in response.content:
    if block.type == "tool_use":
        print(f"Tool: {block.name}, Args: {block.input}")

4. Error Handling — Retry, Backoff & Rate Limits

LLM APIs fail. Your code must assume failure and handle it gracefully. This is what separates production code from a demo.

The most common errors and how to handle them:

  • 429 Rate Limit — You hit the provider's rate limit. Back off, wait, retry.
  • 401 Unauthorized — Your API key is wrong or missing. Don't retry — fix the key.
  • 400 Bad Request — Your messages format is wrong. Don't retry — fix the message.
  • 500 Server Error — The provider had a hiccup. Retry with backoff (2-3 attempts).
  • Timeout — The request took too long. Set a reasonable timeout and retry.
  • Token limit — Your request exceeded context window. Truncate or summarize messages.

⏳ Exponential Backoff with Retry

retry_demo.py 
import time
import random
from openai import OpenAI, RateLimitError, APITimeoutError, APIStatusError

client = OpenAI()

def call_with_retry(messages, max_retries=3, base_delay=1):
    for attempt in range(max_retries):
        try:
            response = client.chat.completions.create(
                model="gpt-4o-mini",
                messages=messages,
                timeout=30,         # ← set a timeout!
            )
            return response

        except RateLimitError:
            delay = base_delay * (2 ** attempt) + random.uniform(0, 0.5)
            print(f"Rate limited. Retrying in {delay:.1f}s (attempt {attempt+1})")
            time.sleep(delay)

        except (APITimeoutError, APIStatusError) as e:
            if e.status_code == 500:
                delay = base_delay * (2 ** attempt)
                print(f"Server error. Retrying in {delay:.1f}s (attempt {attempt+1})")
                time.sleep(delay)
            else:
                raise  # Don't retry 400s or 401s

    raise Exception("Max retries exceeded")

Rule of thumb: Retry 429s and 5xx errors. Never retry 400s or 401s — they won't work on retry and you'll burn through your rate limit for nothing. Always include jitter (random delay) to avoid the "thundering herd" problem.

💰 Tracking Token Costs

Every API response includes usage info — prompt_tokens, completion_tokens, total_tokens. Track these! Your AI engineer salary depends on not burning through budgets.

cost_tracker.py 
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=messages,
)

usage = response.usage
print(f"Prompt tokens: {usage.prompt_tokens}")
print(f"Completion tokens: {usage.completion_tokens}")

# Cost calculation (GPT-4o-mini)
cost = (usage.prompt_tokens / 1_000_000) * 0.15 \
     + (usage.completion_tokens / 1_000_000) * 0.60
print(f"Cost: ${cost:.6f}")

5. API Design & Authentication Patterns

The pattern is the same for every provider. Learn it once, apply everywhere. The specifics change (endpoint URL, key name, message format) but the architecture is identical.

The universal pattern for calling any LLM API:

  1. Load API key from environment variable (never hardcode!)
  2. Create a client with the key
  3. Build your messages payload
  4. Send the request with timeout and retry logic
  5. Parse the response (or stream it)
  6. Track token usage and errors
.env 
# Never commit this file! Add .env to .gitignore
OPENAI_API_KEY=sk-proj-xxxxxxxxxxxx
ANTHROPIC_API_KEY=sk-ant-xxxxxxxxxxxx

🚨 Critical: Never hardcode API keys in your source code. Use environment variables (os.getenv("OPENAI_API_KEY")) or a .env file loaded with python-dotenv. Add .env to .gitignore. If you commit a key, rotate it immediately.

⚠️ Pitfalls — Avoid These

These mistakes cost beginners time, money, and API access. Read them before you touch the exercise.

🔑

Hardcoding API Keys

Committing a key to GitHub means rotating it within minutes — or someone uses your $200 credit on crypto spam. Always use .env + .gitignore.

⏱️

Not Handling Rate Limits

A 429 error without retry logic means your app fails silently for users. Always implement exponential backoff with jitter.

🔄

No Retry Logic

LLM APIs have transient failures (5xx, timeouts). Without retries, you lose completions that would have worked 2 seconds later.

📡

Assuming Streaming Is Optional

Users expect instant feedback. Without streaming, your app feels broken — they stare at a spinner for 10 seconds wondering if it crashed.

💰

Ignoring Token Costs

One runaway loop in production can cost hundreds of dollars. Always log token usage and set spending limits from day one.

📏

Wrong Message Format

OpenAI and Anthropic have different message structures. Sending OpenAI's format to Anthropic (or vice versa) will always fail with a 400.

Exercise 4.1

Multi-LLM Client with Streaming & Error Handling

Build a Python script that calls both the OpenAI and Anthropic APIs with the same prompt, streams both responses, handles rate limits with exponential backoff, and writes the results to a JSONL file with token counts and latency.

1. Create a script multi_llm_client.py that loads API keys from environment variables
2. Implement a call_openai() function with streaming, retry (3 attempts), and token tracking
3. Implement a call_anthropic() function with streaming, retry (3 attempts), and token tracking
4. Call both with the same prompt (e.g., "Explain what an AI engineer does in 3 paragraphs")
5. Write results to llm_results.jsonl — one JSON object per provider with: provider, model, prompt_tokens, completion_tokens, total_tokens, latency_ms, content, status
6. If one provider fails (e.g., you hit a rate limit), the other should still complete — graceful degradation

✅ Success Criteria

  • Script runs end-to-end without errors
  • Both OpenAI and Anthropic responses are streamed to the console in real-time
  • If one API fails, the script continues and records the failure — the other API still returns a result
  • llm_results.jsonl contains valid JSON with token counts and latency for each attempt
  • Token costs are logged to console for each call
  • No hardcoded API keys — everything comes from environment variables or .env
  • The script is committed to your GitHub repo (with .env in .gitignore)
