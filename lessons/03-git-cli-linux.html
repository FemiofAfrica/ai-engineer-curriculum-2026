📦 Git — Version Control Core

Git is the backbone of every professional software project. These are the commands you'll use daily.

git-commands.sh
# ——— Starting a project ——— $ git init # Initialize a new repo $ git clone https://github.com/user/repo.git # Clone an existing repo # ——— Daily workflow ——— $ git status # See what changed $ git add main.py # Stage a file $ git add -A # Stage all changes $ git commit -m "feat: add rate limiting to API client" # Commit with message # ——— History & inspection ——— $ git log --oneline --graph --all # Visual commit history $ git diff # Unstaged changes $ git diff --staged # Staged changes $ git show HEAD # Show latest commit # ——— Branching ——— $ git branch # List branches $ git checkout -b feat/add-embeddings # Create & switch branch $ git switch main # Switch to main # ——— Merging & rebasing ——— $ git merge feat/add-embeddings # Merge branch into current $ git rebase main # Rebase current branch onto main $ git rebase -i HEAD~3 # Interactive rebase (squash, reword) # ——— Stashing ——— $ git stash # Save uncommitted work temporarily $ git stash pop # Restore stashed work # ———— Remotes ——— $ git remote -v # List remotes $ git push -u origin feat/add-embeddings # Push branch & set upstream $ git pull # Fetch + merge latest
💡 Conventional Commits: Use feat: for new features, fix: for bug fixes, docs: for documentation, refactor: for code restructuring, chore: for maintenance. This format is standard across open source and industry.

🖥️ CLI — Command Line Tools Essential

These tools are your daily drivers as an AI engineer. Master pipes and you can compose any data pipeline.

terminal
# ——— curl — Test APIs ——— $ curl https://api.openai.com/v1/models \ -H "Authorization: Bearer $OPENAI_API_KEY" $ curl -X POST https://api.example.com/chat \ -H "Content-Type: application/json" \ -d '{"prompt": "Hello"}' # ——— jq — Parse JSON ——— $ curl -s https://api.github.com/repos/user/repo | jq '.stargazers_count' $ cat data.json | jq '.[] | {name, email}' # ——— grep / rg — Search ——— $ grep -rn "OPENAI_API_KEY" . --include="*.py" $ rg "rate_limit" --type py # rg is 10x faster than grep # ——— Pipes & Redirection ——— $ python train.py 2>&1 | tee training.log # Log stdout + stderr $ gh pr list --json number,title | jq '.[] | .title' # Pipe chain example # ——— Aliases ——— $ alias ll="ls -la" $ alias gst="git status" $ alias glog="git log --oneline --graph --all" $ # Add these to ~/.zshrc or ~/.bashrc to make them permanent

🐧 Linux — The OS You'll Deploy To Foundation

Most AI infrastructure runs on Linux. Here's what you need to navigate and debug a production server.

terminal
# ——— Filesystem hierarchy ——— $ pwd # Where am I? $ ls -la /home/project # List files with details $ du -sh /var/log # Directory size $ df -h # Disk space # ——— Permissions ——— $ chmod +x deploy.sh # Make script executable $ chmod 600 ~/.ssh/id_rsa # Private key (owner rw only) $ chown -R deploy:deploy /app # Change owner $ ls -la # See permissions: drwxr-xr-x # ——— Environment variables ——— $ export OPENAI_API_KEY="sk-..." $ echo $OPENAI_API_KEY $ source .env # Load vars from file $ env | grep API # Show env vars matching API # ——— /proc & system inspection ——— $ cat /proc/cpuinfo | grep "model name" | head -1 $ cat /proc/meminfo | grep MemTotal $ free -h # Memory usage $ nvidia-smi # GPU status (AI-specific) # ——— systemd basics ——— $ systemctl status my-app # Check service status $ journalctl -u my-app -f # Follow service logs $ systemctl restart my-app # Restart a service

⚙️ Process Management Production Skill

Knowing how processes work is the difference between "it stopped when I closed the terminal" and "it runs forever."

terminal
# ——— Foreground/background ——— $ python train.py # Runs in foreground (Ctrl+C to stop) $ python train.py & # Runs in background $ jobs # List background jobs $ fg %1 # Bring job 1 to foreground # ——— Signals ——— $ kill -15 PID # SIGTERM — graceful shutdown $ kill -9 PID # SIGKILL — force kill (last resort) $ kill -2 PID # SIGINT — Ctrl+C equivalent # ——— Keep alive after logout ——— $ nohup python train.py & # Immune to hup signal $ disown %1 # Remove job from shell's table # ——— tmux (terminal multiplexer) ——— $ tmux new -s train-session # Create named session $ # Inside tmux: Ctrl+b d → detach $ tmux attach -t train-session # Reattach later $ tmux ls # List sessions # ——— Cron (scheduled tasks) ——— $ crontab -e # Edit cron jobs $ # Example: run backup daily at 3am $ # 0 3 * * * /home/user/scripts/backup.sh

⚠️ Pitfalls — Avoid These Read First

These mistakes cost every engineer time. Read them before you start the exercises.

🔥 Merge Conflicts
When two people edit the same line of a file, git can't auto-merge. Don't panic — resolve manually: <<<<<<< marks your version, ======= separates, >>>>>>> marks theirs. Edit the file, remove the markers, then git add and git commit.
🚨 Force Push on Shared Branches
git push --force rewrites history. If someone else has pulled your branch, their history diverges. Use git push --force-with-lease instead — it refuses if someone else has pushed new commits.
🧨 Forgetting to source .env
Your app runs fine locally but crashes in production. 90% chance you forgot to load .env. Always check: echo $API_KEY before running. Use export $(grep -v '^#' .env | xargs) to load all vars at once.
💀 Closing Terminal = Killing Process
A process started with python app.py dies when you close the shell. Use nohup, tmux, or a systemd service for anything that needs to survive logout. nohup python app.py & is your emergency escape.
🔑 Committing Secrets
Accidentally committed an API key? Even if you delete it in the next commit, it's in the git history forever. Use git-secrets, a pre-commit hook, or .env files to prevent this. If you do leak one, rotate the key immediately.
📏 git add . Without Reviewing
git add . stages everything — including debug logs, compiled files, and secrets. Always run git status and git diff first. Use .gitignore aggressively.

Exercise 1

Git Workout — Local Repo & Branches

Create a local git repository, make commits, experiment with branches, and practice resolving a merge conflict.

  1. Create a new directory ~/workout-repo, initialize git
  2. Create a README.md, stage it, commit with message docs: initial readme
  3. Create a branch feature/greeting and switch to it
  4. Add a hello.py that prints "Hello from feature branch"
  5. Commit with message feat: add greeting script
  6. Switch back to main, edit README.md to add a different line
  7. Commit on main, then merge feature/greeting into main
  8. View the commit graph: git log --oneline --graph --all
  9. Challenge: Create a second branch, intentionally create a conflict, then resolve it manually
✅ Success Criteria
  • Multi-branch history visible in git log --graph
  • Merged feature branch (or a merge commit exists)
  • Merge conflict resolved cleanly (no conflict markers remain)
  • Conventional commit messages used throughout
Exercise 2

CLI Pipeline — Inspect & Transform an API Response

Use curl + jq + grep to inspect a live API, extract meaningful data, and log it.

  1. Fetch the Hacker News top stories: curl -s https://hacker-news.firebaseio.com/v0/topstories.json
  2. Use jq to count how many stories are returned (jq length)
  3. Fetch the first story's details: curl -s https://hacker-news.firebaseio.com/v0/item/FIRST_ID.json
  4. Use jq to extract the title, score, and url into a clean format
  5. Use grep or rg to find stories mentioning "AI" or "LLM" from the top 50 IDs
  6. Pipe the results into a file ai-stories.log
  7. Write a one-liner that does the entire pipeline in a single command using |
✅ Success Criteria
  • ai-stories.log exists with real story data
  • jq queries used filters (.[], select(), {title, score})
  • Single-command pipeline demonstrated
  • Pipes (|) used to chain commands
Exercise 3

Real-World Git — Open a Pull Request

Clone a public repo, create a meaningful change on a new branch, commit properly, push, and open a PR via gh CLI.

  1. Find a public repo by @cyrilXBT or any small open source Python project
  2. Fork it via GitHub UI or gh repo fork
  3. Clone your fork locally: gh repo clone YOUR_USER/REPO
  4. Create a branch: git checkout -b improve/docs
  5. Make a meaningful change — fix a typo in README, improve a docstring, add a CLI flag, or write a better error message
  6. Commit with conventional format: docs: fix typos in README or feat: add --verbose flag to CLI
  7. Push the branch: git push -u origin improve/docs
  8. Open a PR: gh pr create --title "docs: fix typos in README" --body "..."
  9. Verify: gh pr view --web or gh pr list
✅ Success Criteria
  • gh pr list shows your PR with correct title and branch
  • gh pr diff shows only your intended changes (no stray files)
  • Commit message follows conventional format
  • The PR branch is pushed to your fork on GitHub
  • The change is genuinely useful (not just a whitespace edit)
